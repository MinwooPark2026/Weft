# script-to-conti — 대본을 Weft 이중 트랙 콘티로

산문 대본 한 편을 **Weft 콘티 `CONTI.md`** 로 바꾼다. 이 콘티는 `weft conti` 가
바로 파싱하는 정본 입력이다.

대본은 **사용자가 직접 쓴 완성 원고**다. 주제는 불문 — 음악사 분석, 기후 사고실험,
수학과 철학의 비교, 무엇이든 들어온다. 이 스킬은 **내용을 새로 쓰거나 사실을 고치는
단계가 아니다**: 대본의 문장·주장·톤을 그대로 존중하고, 시각 연출(비트 분할·시각 밀도·
프롬프트)만 설계한다. 대본에 의심스러운 부분이 있으면 고치지 말고 사용자에게 표시해서 묻는다.

- **이 스킬 = 앞단**: 대본 → 비트 분할 + 시각 밀도 결정(▶/↓/▦/↺/⏸/❝) + shot별 영어 프롬프트.
- **Weft 앱(CLI) = 뒷단**: `conti`(정본화·검증) → `tts`(나레이션) → `images`(생성) → `pick`(후보 선택) → `capcut`(드래프트).

> ⚠️ 이건 **1:1(한 컷=한 이미지) 콘티가 아니라 이중 트랙 콘티**다. 한 행 = 한 컷이 아니라 **한 행 = 한 나레이션 비트**이고, 시각은 `시각(shot)` 칸으로 분리돼 여러 비트를 덮거나(홀드) 한 비트에 여러 장(몽타주)이 붙는다.

## 먼저 읽을 것

이 스킬 문서 하나로 콘티를 만들 수 있다(아래 §출력 형식에 표 스키마가 다 있음). 함께 보면 좋은 것:

- **gold 예시**: [`example/CONTI.md`](../../../example/CONTI.md) — 검증 0 에러로 통과한 실제 콘티. 구조를 이걸로 모사한다.
- [`STYLE_GUIDE.md`](../../../STYLE_GUIDE.md) — 이미지 일관성 설정·교체. 모든 이미지 프롬프트가 상속할 `Style:` 문장은 여기서 정한다.

## 핵심 원칙

- **나레이션이 스파인**: 대본을 문장(beat) 단위로 세운다. 시각은 그 옆에 붙는다.
- **시각 밀도를 명시적으로**: 비트마다 `▶ 새 그림 / ↓ 홀드 / ▦ 몽타주` 중 하나. `↓`로 이어지는 동안은 그림이 안 바뀐다 = 그만큼 덜 골라도 된다. **`▶`를 줄이는 게 핵심 가치**(이미지 수 = `▶`+`▦`슬롯).
  - "한 그림에 여러 설명" → **홀드(↓)**: 도식 1장이 4~6문장을 덮음.
  - "한 문장에 여러 그림" → **몽타주(▦)**: 나열·열거 문장에.
  - 수미상관/콜백 → **재사용(↺)**: 생성·픽 0.
- **조용히 100비트를 한 방에 만들지 마라.** 막(act) 단위로 분할안을 표로 보여주고 "여긴 더 홀드 / 여긴 몽타주" 피드백을 받아 고친 뒤 다음 막으로. 반복 편집 루프가 이 스킬의 가치다.

## 출력 형식 — CONTI.md (파서가 읽는 정확한 스키마)

제목은 첫 `# ` 줄. 머리말(분량·스타일 바이블·범례)은 불릿/인용으로. 그다음 **두 개의 markdown 표**:

### 1) 콘티 표 (첫 표, 첫 컬럼 헤더 = `beat`)

```markdown
| beat | 시각(shot) | 시간 | 나레이션 (TTS) | 자막 | 모션·메모 |
|------|-----------|------|----------------|------|-----------|
| b001 | ▶ s01_xxx | 0:00~0:06 | 문장 원문… | 화면 자막 | zoom_in, 메모 |
| b002 | ↓ | 0:06~0:12 | 다음 문장… | 자막 | (홀드) |
| b003 | ▦ s10_a / s11_b / s12_c | … | 나열 문장… | a · b · c | 각 zoom_in |
| b004 | ⏸ 2초 | | | | 호흡 |
| b005 | ❝ s20_quote | … | 인용 낭독… | "핵심 인용" | 타이포 카드 |
| b006 | ↺ s01_xxx | … | 콜백 문장… | (수미상관) | zoom_in |
```

`시각(shot)` 토큰: `▶`(새 shot) · `↓` 또는 빈칸(홀드) · `▦ a / b / c`(몽타주, `/`로 구분) · `↺ s..`(재사용) · `⏸ N초`(정적) · `❝ s.._quote`(텍스트카드) · `⤴ s..`(빌드업) · `🏷 title`/`🏁 end`(화면요소). 시간 칸은 비워둔다(TTS가 역기입). 셀 안 리터럴 `|` 금지(→`/`), 줄바꿈 `<br>`.

### 2) shot 표 (둘째 표, 첫 컬럼 헤더 = `shot id`)

```markdown
| shot id | source_kind | cover | 모션 | 프롬프트 / 문구 |
|---------|-------------|-------|------|-----------------|
| s01_xxx | image | b001~b002 | zoom_in | An English image prompt describing only this shot's subject/composition |
| s10_a | image | b003 | zoom_in | … |
| s20_quote | text_card | b005 | fade | 한국어 문구: "핵심 인용" (generated_project/CARDS.json 으로 렌더) |
```

- `source_kind`: `image`(생성) / `remotion`(React 애니메이션 MP4) / `hyperframe`(HTML/CSS 애니메이션 MP4) / `text_card`(타이포 카드) / `screen_element`(타이틀·엔딩) / `reuse`(→ id) / `clip`(기존 MP4) / `stock_clip`.
- `프롬프트 / 문구`: image는 **영어** 프롬프트(그 shot 고유 피사체·구도만; `Style:`은 자동 상속). text_card는 한국어 문구.

## 이미지 프롬프트 가이드

- **영어**, 한 shot = 한 문장~짧은 문단. 모델 gpt-image-1. 후보 2~3장 변주가 되도록 구체적이되 과하지 않게.
- **생성 이미지엔 문자 금지** (스타일 바이블 규칙). "단어가 개념인" shot(어텐션·라벨·연도·필터 등)은 글자를 부르니, 순수 시각으로 풀고 프롬프트 끝에 "absolutely no text/letters/numbers/labels"를 명시. 글자가 필요하면 **`❝` 텍스트카드**로 분리.
- 도식/은유 반반. 도식은 홀드(↓)에, 사물·은유 나열은 몽타주(▦)에 잘 맞는다.
- **한국어 대본 → 영어 프롬프트 변환**:
  - 같은 피사체가 여러 `▶` shot에 재등장하면 **명사구를 한 번 정해 글자 그대로 복사**해 쓴다
    (예: 모든 shot에서 "a weathered cassette tape player" 고정). 묘사가 흔들리면 그림체가 같아도 다른 물건이 나온다.
  - 한국 고유 소재는 모델이 아는 구체적 영어로 풀어 쓴다(예: "1990년대 길보드" → "a 1990s Korean street
    cassette vendor cart with stacked tapes"). 고유명사만 던지지 말 것.
  - 한국어 은유는 직역하지 말고 **시각 개념으로 번역**한다("마음의 짐" → "a figure carrying an oversized
    translucent boulder"). 은유의 원관념이 아니라 보조관념을 그린다.

## 애니메이션 shot 선택

정적인 은유·삽화는 `image`로 둔다. 숫자가 변하는 그래프, 단계별 도식, 수식 전개, 타임라인 reveal처럼 움직임이 설명 자체인 장면은 `source_kind=remotion` 또는 `source_kind=hyperframe`을 선택한다. `weft animate`가 `SHOTS/<shot>/animation/SPEC.md`와 기대 출력 `SHOTS/<shot>/rendered/clip.mp4`를 알려주면, AI agent가 Remotion/HyperFrame으로 그 MP4를 만든다. 렌더된 MP4는 `weft ffmpeg`, `weft capcut`, `weft fcpxml`에 일반 clip처럼 들어간다.

## 텍스트 카드 (❝) 처리

`❝` shot의 실제 온스크린 문구는 `generated_project/CARDS.json` 에 `{shot_id: "문구\n여러 줄"}` 로 넣는다(빌드 시 Pillow가 한국어 타이포 PNG로 렌더). 콘티의 `자막`/`프롬프트 / 문구` 칸에 의도를 적어두고, CARDS.json에 최종 문구를 확정.

> ⚠️ **렌더러는 중앙 정렬 타이포 한 덩어리만 그린다.** 순위표·막대그래프·표·강조 흐림 같은
> 레이아웃은 `❝`로 불가능하다 — 그런 장면은 `image`(정적 도식)나 `remotion`/`hyperframe`(움직이는
> 도식)으로 보낸다. `❝`는 인용·정의·한 줄 선언처럼 "글자 자체가 화면"인 곳에만.

## API 키 판단 규칙

Weft로 영상을 만들 때 API 키의 문자열 모양만 보고 placeholder라고 판단하지 마라. Typecast 키는 `__`로 시작할 수 있고 그래도 정상 키일 수 있다. `TYPECAST_API_KEY`/`TYPECAST_VOICE`가 비어 있거나 `TTS_PROVIDER=stub`이 명시된 경우만 stub/placeholder 운용으로 본다. 그 외에는 `weft tts`를 실행해서 Typecast HTTP 응답, 생성된 WAV, sidecar metadata의 `provider=typecast`로 실제 작동 여부를 판단한다. secret 값은 로그나 답변에 출력하지 않는다.

## 절차

1. **입력 파악**: 대본 .md 경로 + 목표 길이 + 스타일(프로젝트 STYLE.txt, 없으면 내장 DEFAULT_STYLE — STYLE_GUIDE.md 참고) + 출력 위치(`<project>/CONTI.md`).
2. **분량 산정**: 한국어 나레이션은 **약 4.8자/초**. 대본 글자수 ÷ 4.8 = 예상 길이(초). 비트는 평균 6~9초이므로 비트 수 ≈ 길이÷7, `▶`(새 이미지) 예산은 **비트 수의 1/3~1/2**에서 시작한다. **15분(비트 ~130개) 이상 긴 대본은 반드시 막 단위로 나눠 진행**하고, 막마다 콘티를 이어 붙이며 중간 검증한다.
3. **막 단위로 비트 분할 + 시각 밀도 결정**. 막마다 표로 보여주고 피드백 받아 고침.
4. **shot별 영어 프롬프트 작성**(스타일 바이블 상속, 문자 금지, 재등장 피사체 어휘 고정).
5. **대본 점검(수정 아님)**: 숫자·연도·고유명사 중 의심스러운 것, 사실처럼 들리는 추측 문장을 **사용자에게 표시해 확인**받는다. 대본을 임의로 고치지 않는다. 상상/사고실험 구간은 §주제 유형별 가이드대로 "가정임이 화면에 드러나는지"를 점검한다.
6. **CONTI.md + CARDS.json 작성** → 검증:
   ```bash
   cd <project> && weft conti        # ./CONTI.md 를 읽어 ./generated_project 로 씀
   ```
   **검증 오류 0** 이어야 한다(partition 불변식: 모든 비트가 정확히 한 번 덮임). 오류 나면 `시각` 칸/표 스키마를 고친다. `❝` shot이 있으면 `generated_project/CARDS.json`에 최종 문구까지 넣어야 완료다(빠뜨리면 shot id 문자열이 그대로 화면에 렌더된다).
7. **다음**: AI agent는 보통 `weft tts` → `weft images` → `weft animate` → `weft ffmpeg` 순서로 단계별 확인한다. `weft all`은 사람이 빠르게 MP4를 뽑는 shortcut이다. MP4가 마음에 안 들거나 NLE 편집이 필요할 때만 `weft capcut` 또는 `weft fcpxml`을 추가 실행한다.

## 주제 유형별 가이드

대본 주제는 불문이다. 자주 만나는 유형별 연출 요령:

- **분석·설명형** (역사, 과학, 기술): 기본형. 도식은 홀드(↓)로 길게, 사례 나열은 몽타주(▦)로.
- **사고실험·상상형** (가상 시나리오, "만약 ~라면"): **사실 구간과 가정 구간을 화면에서 구분**한다.
  가정으로 진입하는 비트에 `❝` 카드("만약 오호츠크해의 습도가 절반이 된다면 —")나 자막 표기를 두고,
  모션·메모에 톤 전환을 적는다. 시청자가 시뮬레이션 결과를 관측 사실로 오해하지 않게 하는 것이 연출 책임이다.
- **비교·대조형** (두 사상/이론의 병렬 분석): 짝이 되는 shot은 **id를 짝으로 명명**(`s12_math_grid` /
  `s12_relig_grid`)하고 프롬프트의 구도 어휘를 맞춰 시각적 대구를 만든다. 한 문장 안의 대비는
  `▦` 2슬롯 몽타주가 분할 화면 역할을 한다. 교차 반복 구간은 ↺ 재사용으로 리듬을 만든다.
- **음악·소리 중심** (음악사, 청취 환경 분석): BGM·음원 트랙은 아직 Weft 스키마에 없다. 보유하거나
  라이선스가 확보된 영상·음원만 `source_kind=clip`으로 넣고, 그 외에는 소리를 **시각으로 번역**한다
  (스피커·기기·파형·공간의 변화). 시대별 청취 기기·환경은 몽타주 소재로 특히 좋다.

## 톤

**대본의 톤을 따른다** — 콘티가 대본의 장르를 바꾸지 않는다. 공통 원칙: 깊이 파고들되 차분·명료, 비전공자도 안 지루하게. 정적(⏸)·톤 변화·시각 회상(↺)으로 리듬을 만든다 — 단순 나레이션 나열이 아니라 편집의 차원(홀드·몽타주·정적·수미상관)으로 의미를 쌓는다.
